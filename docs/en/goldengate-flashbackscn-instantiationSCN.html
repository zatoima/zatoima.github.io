Instantiation SCN When Using FLASHBACK_SCN During expdp

Published on 2019/03/30 üìñ 1 min read
Oracle GoldenGate

Introduction

GoldenGate has a feature called Instantiation SCN. Using this feature, you can easily perform a zero-downtime initial migration using Data Pump.

For details, please refer to the following Qiita article which explains this well:

Easy Initial Migration Using Instantiation SCN - Qiita https://qiita.com/kurouuuron/items/118afdc6b6d15d46e7dd

A GoldenGate feature for initial migration that leverages integration with Oracle Data Pump.

In short, it makes zero-downtime initial migration using Data Pump easy.

More specifically, it automates considerations that are normally required for initial migration, such as Flashback_scn and Handlecollisions.

In practice, when you run expdp with Data Pump, it internally records up to which SCN each table’s data has been applied. When impdp is run, the Replicat automatically applies data starting from after that SCN.

As mentioned in this article, when add trandata is executed with GoldenGate 12.2 or later, preparescn is applied by default. This means that when you run expdp with Data Pump, it internally records up to which SCN each table’s data has been applied.

Environment Preconditions

The behavior described here applies when add trandata preparescn none is specified.

Note that the behavior may differ if a value other than none is set.

Behavior When add trandata preparescn none is Set

As stated in the manual , in this case, preparation for instantiation is not performed.

Check Instantiation SCN

Since preparescn none was used, there is no instantiation SCN in this case.

SQL> select source_database,source_object_owner,source_object_name,instantiation_scn from  DBA_APPLY_INSTANTIATED_OBJECTS where SOURCE_OBJECT_NAME='T10';

no rows selected

When FLASHBACK_SCN is Used During expdp

This is the main topic: even when trandata is set with preparescn none, it appears that the instantiation SCN is set during expdp when FLASHBACK_SCN is specified.

Export Command

Specify the FLASHBACK_SCN for the desired point in time.

[oracle@xxxxxxxx1p ~]$ expdp ggtest/xxxxxx DIRECTORY=homedir DUMPFILE=t10.dmp TABLES=t10 CONTENT=DATA_ONLY REUSE_DUMPFILES=YES FLASHBACK_SCN=25587768

Export: Release 11.2.0.4.0 - Production on Sun Mar 3 16:59:04 2019

...
. . exported "GGTEST"."T10"                              288.6 KB   10000 rows
...
Job "GGTEST"."SYS_EXPORT_TABLE_01" successfully completed at Sun Mar 3 16:59:08 2019 elapsed 0 00:00:02

When importing a dump file that was exported with FLASHBACK_SCN, INSTANTIATION_SCN is set in DBA_APPLY_INSTANTIATED_OBJECTS.

SQL> select source_database,source_object_owner,source_object_name,instantiation_scn from  DBA_APPLY_INSTANTIATED_OBJECTS where SOURCE_OBJECT_NAME='T10';

Summary

Even when preparescn none is set with add trandata, using FLASHBACK_SCN during expdp results in the instantiation SCN being set.

This behavior is documented in the Streams manual:

https://docs.oracle.com/cd/E16338_01/server.112/b61352/instant.htm#CIHBDFJE

During export, Oracle Data Pump automatically uses Oracle Flashback to ensure consistency of exported data and procedural operations for each database object at a specific point in time. When performing instantiation in an Oracle Streams environment, some degree of consistency is required. The Data Pump export utility ensures this consistency for Oracle Streams instantiation.

If you use the export dump file for purposes other than Oracle Streams instantiation, and those purposes have stricter consistency requirements than what the default Data Pump export provides, you can use the Data Pump export utility parameters FLASHBACK_SCN or FLASHBACK_TIME for the Oracle Streams instantiation. For example, stricter consistency may be required if the export includes objects with foreign key constraints.
