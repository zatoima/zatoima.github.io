Redshift Spectrum Concurrency Performance

Published on 2022/01/29 📖 1 min read
AWS Redshift

Here are some notes on how the concurrency performance of Redshift Spectrum is determined.

1. Number of Slices

This factor is an important concept not only for Spectrum but also for standard Redshift.

The number of slices is primarily determined by the instance type (instance size). Each slice is allocated a portion of the node’s memory and disk capacity and processes a portion of the workload assigned to the node. The leader node manages the distribution of data to slices and distributes the workload of queries or other database operations to slices, with each slice handling its own portion of the workload. As shown below, each node contains multiple slices.

Reference: Data Warehouse System Architecture - Amazon Redshift

The number of slices is predetermined by the node size as the default number of slices per node.

Amazon Redshift Clusters - Amazon Redshift - Node Type Details

For RA3, ra3.xlplus has “2”, ra3.4xllarge has “4”, and ra3.16xlarge has “16”.

There is a potential pitfall to be aware of during operations: depending on how you resize the node size, the number of slices may not change.

[Amazon Redshift Clusters - Amazon Redshift - Node Type Details](Amazon Redshift Clusters - Amazon Redshift )

  • The number of slices per node is the default number of slices into which a compute node is partitioned when you create or resize a cluster using classic resize.
  • When you resize a cluster using elastic resize, the number of slices per node might change. However, the total number of slices across all compute nodes in the cluster remains the same after elastic resize.
  • When you create a cluster using a restore from snapshot operation, if you change the node type, the number of slices in the resulting cluster may differ from the original cluster.

2. Concurrency

There is a maximum of 10 concurrent requests per query per slice.

  • The concurrency varies depending on how many queries are running simultaneously
  • The allocated concurrency limits the number of threads scanning S3

*Note: The word “maximum” here is likely an important caveat. Also note that, as far as I can tell, there is no documentation in the manual about the concurrency between the Spectrum layer and S3. While this is presumably something that is abstracted away from the user, the exact specification is not explicitly stated.

Reference: 12 Best Practices for Amazon Redshift Spectrum | Amazon Web Services Blog

Spectrum Performance

Combining the discussions about “1. Number of Slices” and “2. Concurrency,” the relationship can be summarized as follows.

How to Check Spectrum Concurrency

Check the SVL_S3QUERY_SUMMARY table. It contains statistics such as the number of files processed per slice and parallel requests, so you can verify concurrency here.

SVL_S3QUERY_SUMMARY - Amazon Redshift

Column Name Description
files The number of files processed by this Redshift Spectrum query. A small number of files limits the benefits of parallel processing.
files_max The maximum number of files processed by a single slice.
files_avg The average number of files processed by a single slice.
splits The number of splits processed in this segment. The number of splits processed by this slice. For example, if a splittable data file is large (such as a data file exceeding approximately 512 MB), Redshift Spectrum splits the file into multiple S3 requests and attempts parallel processing.
splits_max The maximum number of splits processed by this slice.
splits_avg The average number of splits processed by this slice.
total_split_size The total size of all splits processed.
max_split_size The maximum size of a processed split (in bytes).
avg_split_size The average size of a processed split (in bytes).
total_retries The total number of retries for individual processed files.
max_retries The maximum number of retries for a processed file.
max_request_duration The longest time for an individual file request (in microseconds). Long-running queries may indicate a bottleneck.
avg_request_duration The average time for file requests (in microseconds).
max_request_parallelism The maximum number of parallel requests for a single slice in a Redshift Spectrum query.
avg_request_parallelism The average number of parallel requests for a single slice in a Redshift Spectrum query.

How to Maximize Spectrum Concurrency Performance

As described above, you need to verify that the number of split files on S3 and the concurrency are at an optimal balance.

12 Best Practices for Amazon Redshift Spectrum | Amazon Web Services Blog

A simple calculation would be: total number of files <= concurrency per query (e.g., 10) * number of slices in the cluster. However, increasing the number of nodes in the cluster does not necessarily improve performance. Determine the optimal number of cluster nodes as follows: First, check how many files are contained in the Redshift Spectrum external table. Then, gradually increase the cluster size (meaning increase the number of slices in the cluster) and find the point at which increasing the number of nodes no longer improves performance. The optimal Redshift cluster size for that node type is the point beyond which no further performance improvement occurs.

As a brute-force approach, you can increase the degree of parallelism further by setting up multiple Redshift clusters and running queries across them. I believe this is one of the advantages of Spectrum, where storage and compute are separated. You can set up multiple clusters and query the same S3 files from each cluster.

Supplement: Redshift Spectrum Concurrency Scaling

Redshift Spectrum itself is also eligible for Concurrency Scaling. Previously, Spectrum was not eligible for Concurrency Scaling. The manual description is somewhat unclear, but there is a mention in the Forums. (*Note: I have not actually tested this.)

  • Using Concurrency Scaling - Amazon Redshift
    • Queries that access external resources protected by restricted network or virtual private cloud (VPC) configurations are not supported.
  • AWS Developer Forums: Amazon Redshift Announcements ...
    • Redshift Spectrum: UNLOAD and queries that use Spectrum to reference external tables are now eligible for Concurrency Scaling
