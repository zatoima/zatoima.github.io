Aurora PostgreSQL DB Audit Methods (Database Activity Streams or pgaudit?)

Published on 2022/04/16 📖 3 min read
AWS Aurora PostgreSQL

Features

pgaudit

  • A PostgreSQL extension

  • Output is mixed into PostgreSQL logs (also output to CloudWatch in the case of Aurora)

  • You can narrow down audit targets by specifying session audit/object audit/DML/DDL

  • Sample of captured information

  • A usage note is that there is a performance impact. The following is from GitHub:

    • Depending on settings, it is possible for pgAudit to generate an enormous volume of logging. Be careful to determine exactly what needs to be audit logged in your environment to avoid logging too much.

      For example, when working in an OLAP environment it would probably not be wise to audit log inserts into a large fact table. The size of the log file will likely be many times the actual data size of the inserts because the log file is expressed as text. Since logs are generally stored with the OS this may lead to disk space being exhausted very quickly. In cases where it is not possible to limit audit logging to certain tables, be sure to assess the performance impact while testing and allocate plenty of space on the log volume. This may also be true for OLTP environments. Even if the insert volume is not as high, the performance impact of audit logging may still noticeably affect latency.

  • Reference link

https://github.com/pgaudit/pgaudit

Database Activity Streams

  • Can store near-real-time data

  • Enables separation of duties

    • For example, if there is a DBA and a security administrator, only the security administrator can review the audit logs

  • Both synchronous and asynchronous modes exist. Each has its pros and cons, making the choice difficult.

    • Asynchronous mode - When a database session generates an activity streaming event, the session immediately returns to normal activity. The activity streaming event becomes a persistent record in the background. If an error occurs in the background task, an RDS event is sent. This event indicates the start and end of the time window during which activity streaming event records may have been lost.

      In asynchronous mode, database performance takes priority over activity streaming accuracy.

    • Synchronous mode - When a database session generates an activity streaming event, the session blocks other activities until the event is persisted. If the event cannot be persisted for any reason, the database session returns to normal activity. However, an RDS event is sent indicating that activity streaming records may have been lost for a period of time. When the system returns to a healthy state, a second RDS event is sent.

      In synchronous mode, activity streaming accuracy takes priority over database performance.

  • Additional charges are required for Kinesis streaming

  • Refer to the manual for the information that can be captured

    • https://docs.aws.amazon.com/ja_jp/AmazonRDS/latest/AuroraUserGuide/DBActivityStreams.Monitoring.html

  • Data can be output to 3rd party monitoring tools or S3 via Kinesis. When output to S3, the data is in JSON format, so visualization or SQL-based analysis would be required.

    • The flow would be: Aurora - DAS - Kinesis Stream - Kinesis Firehose - Decrypt with Lambda - Output to S3
      • https://dev.classmethod.jp/articles/aurora-postgresql-audit-with-data-activity-stream/

  • When using only AWS services, additional consideration for Athena or QuickSight for analysis is required.

Conclusion

It depends on the requirements, but if there are no specific constraints, the following guidelines would be appropriate:

  • When to use pgaudit

    • When the additional cost of DAS feels too high, when the specifications of synchronous/asynchronous modes are unacceptable, when you want to adopt the same audit method as on-premises PostgreSQL due to the additional configuration considerations when using only AWS, when you want to use pgaudit which has a proven track record with on-premises PostgreSQL, or when you only want to audit specific important tables

  • When to use DAS (Database Activity Streams)

    • When you want to pursue real-time capabilities, when you need separation of duties, when capturing all table information with pgaudit causes performance issues and you want to use DAS asynchronous mode for lower overhead, or when you want to capture various metrics

Appendix: Simple Performance Benchmark

Since the performance differences are also a concern, I ran a benchmark using pgbench. This is an N=1 result, so it may not be very reliable. The usual disclaimer applies: please evaluate with your actual workload.

The results are from running pgbench with 32 clients continuously for 300 seconds.

#ベンチマークテスト
create database pgbench;
pgbench -i -s 1000 -U postgres -h aurorapgsqlv1.cluster-xxxxxxx.ap-northeast-1.rds.amazonaws.com -d pgbench
pgbench -r -c 32 -T 300 -U postgres -h aurorapgsqlv1.cluster-xxxxxxx.ap-northeast-1.rds.amazonaws.com pgbench

Results Summary

Perhaps because the load was low, the results showed almost no degradation except for DAS (synchronous mode) enabled. I recall that enabling pgaudit on on-premises PostgreSQL caused a bit more degradation, but in this case there was almost no degradation.
