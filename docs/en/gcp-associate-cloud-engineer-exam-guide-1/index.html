GCP Associate Cloud Engineer Exam Guide - Setting Up Cloud Solution Environments

Published on 2021/09/05 üìñ 2 min read
GCP

Introduction

In preparation for the GCP Associate Cloud Engineer exam, I review the manuals listed in the exam guide and conduct some hands-on verification to study according to the exam guide. At the major section level, the following 5 topics are in the exam scope:

  1. Setting up a cloud solution environment
  2. Planning and configuring a cloud solution
  3. Deploying and implementing a cloud solution
  4. Ensuring successful operation of a cloud solution
  5. Configuring access and security

The Certification Exam Guide further subdivides these major sections, and each one seems to have specific exam topics broken down as follows.

First, let’s review Setting up a cloud solution environment.

1. Setting up a cloud solution environment

1.1 Setting up cloud projects and accounts. Activities include:

  • Creating a project

    Reference: Creating and Managing Projects | Resource Manager Documentation | Google Cloud

A Google Cloud project is the foundation for creating, enabling, and using all Google Cloud services, including managing APIs, enabling billing, adding and removing collaborators, and managing permissions for Google Cloud resources. For example, if you want to separate production and development environments, you separate these into different projects. In AWS, this would correspond to account separation, where you create IAM users for each. In AWS, environment (account) switching uses IAM Switch Role. Personally, GCP’s approach feels more intuitive.

Console

gcloud command
gcloud projects create PROJECT_ID

  • Assigning users to predefined IAM roles within a project

    Reference: Managing Access to Projects, Folders, and Organizations | Cloud IAM Documentation | Google Cloud

    • Select roles to add from “IAM & Admin” - “IAM”.

    • GCP IAM elements are as follows. Note that although similar terms to AWS are used, the contents are quite different.

      • Reference: FAQ | Cloud IAM Documentation | Google Cloud

      • Members: The target to which permissions are granted. Google accounts, service accounts, Google Groups, G Suite, Cloud Identity domains.

      • Roles: A collection of permissions. When you grant a role to a member, all permissions included in the role are granted.

        • Basic roles, predefined roles, and custom roles exist. Basic roles include owner, editor, and viewer roles that affect all GCP services. Predefined roles are permission sets prepared by GCP. Custom roles are permission sets configured by the user.

        • The manual has this note about basic roles:

          Caution: Basic roles include thousands of permissions across all Google Cloud services. In a production environment, do not grant basic roles unless there are no alternatives. Instead, grant the most restrictive predefined role or custom role that meets your needs.

        • AWS users might confuse this, but GCP roles are equivalent to AWS IAM policies.

      • Policies: Controls who (user) has what (role) permissions on which resource.

        • Similar to bucket policies in AWS.

        • https://cloud.google.com/iam/docs/faq?hl=ja#what_does_an_policy_look_like

  • Managing users in Cloud Identity (manually and automatically)

    • Since it’s a Google service, GCP accounts can use Google accounts. Organizations that can’t create Google accounts, or organizations using G Suite, manage accounts with Cloud Identity.

      Reference: Setup Instructions for GCP Administrators - Cloud Identity Help

      Cloud Identity is an IDaaS (Identity as a Service) solution and an enterprise mobile management (EMM) service. It provides the identity services and endpoint management available in Google Workspace as a standalone service. With Cloud Identity, administrators can centrally manage users, apps, and devices from the Google Admin console.

    • It also supports common Microsoft Active Directory federation.

      • Federating Google Cloud with Active Directory: Overview | Architecture

  • Enabling APIs within projects

    • In GCP, you need to enable the API for the service you want to use. You can enable it from the API library screen or via gcloud.

      • Why is GCP designed to require API enablement?
        • This kind of one-liner is inevitable…
        • One-liner to enable all GCP APIs - Qiita

      Reference: Enabling APIs in a Google Cloud Project | Cloud Endpoints with OpenAPI

    • From the API library:

    • gcloud command:

      gcloud services enable SERVICE_NAME

  • Provisioning Stackdriver workspaces

    • Stackdriver has been integrated into Operations Suite, and provisioning a workspace separately appears to be unnecessary.
    • Operations Suite overview:
      • Cloud Logging
        • Real-time log management and analysis
        • Can ingest application and system log data, as well as custom log data from GKE environments, VMs, and Google Cloud services
      • Cloud Monitoring
        • Built-in large-scale metrics observability
        • View performance, uptime, and overall behavior of applications running in the cloud. Collect metrics, events, and metadata from Google Cloud services, hosted uptime probes, application instrumentation, and various popular application components, visualize in charts and dashboards, and manage alerts.
      • Application Performance Management (APM)
        • Integrates Cloud Trace , Cloud Debugger , Cloud Profiler
        • Cloud Trace
          • Detect performance bottlenecks in production
            • Cloud Trace automatically analyzes all application traces and generates detailed latency reports identifying causes of performance degradation
        • Cloud Debugger
          • Investigate code behavior in production
            • Real-time debugging of applications
        • Cloud Profiler
          • Continuous profiling of CPU and heap to help improve performance and reduce costs

1.2 Managing billing configuration. Activities include:

  • Creating a billing account

    • Requires billing.accounts.create permission

      Reference: Create, Modify, or Close a Self-Serve Cloud Billing Account | Cloud Billing | Google Cloud

  • Linking projects to a billing account

    Reference: Enable, Disable, or Change Billing for a Project | Cloud Billing | Google Cloud

  • Establishing billing budgets and alerts

    Reference: Setting Budget Alerts | Cloud Billing | Google Cloud

  • Setting up billing exports to estimate daily/monthly charges

    • Using the Cloud Billing export to BigQuery feature, you can automatically export detailed Google Cloud billing data (usage, cost estimates, pricing data, etc.) to a specified BigQuery dataset throughout the day. Note that Cloud Billing data before enabling export will not be displayed.

      Reference: Export Cloud Billing Data to BigQuery | Google Cloud

1.3 Installing and configuring the command-line interface (CLI), specifically Cloud SDK (e.g., setting the default project)

  • When the manual says “SDK configuration,” it’s unclear what it refers to, but in English it seems to mean “Configuration” ‚Äî I understand it as something like a profile or settings.

    • Create a configuration:

      gcloud config configurations create [NAME]

    • Activate a configuration:

      gcloud config configurations activate [NAME]

    • List configurations:

      gcloud config configurations list

    • Set configuration properties:

      gcloud config set project [PROJECT]
gcloud config unset project

    • View configuration properties:

      gcloud config configurations describe [NAME]

    • Delete a configuration:

      gcloud config configurations delete [NAME]

    Reference: Managing SDK Configurations | Cloud SDK Documentation | Google Cloud
