## Introduction

You often hear that "properly maintaining metadata improves the quality of SQL generated by AI agents." But how much difference does it actually make? There are few examples that verify this with concrete comparisons.

In this article, I use Snowflake's retail sample data (`RETAIL_SAMPLE_DB`) to compare how the SQL generated by `SNOWFLAKE.CORTEX.COMPLETE` changes before and after adding the following three types of metadata.

| Metadata | Snowflake Feature |
|-----------|--------------|
| Column comments | [`COMMENT ON COLUMN`](https://docs.snowflake.com/en/sql-reference/sql/comment) |
| PII tags | [`CREATE TAG`](https://docs.snowflake.com/en/sql-reference/sql/create-tag) |
| Semantic views | [`CREATE SEMANTIC VIEW`](https://docs.snowflake.com/en/sql-reference/sql/create-semantic-view) |

## Test Environment and Table Structure

I used the star schema in the `RETAIL_SAMPLE_DB.SALES` schema for this verification.

```
FCT_INVENTORY (15,930 rows)
  ├── DATE_KEY    → DIM_DATE
  ├── STORE_KEY   → DIM_STORE (15 stores)
  ├── PRODUCT_KEY → DIM_PRODUCT (59 products)
  └── SUPPLIER_KEY → DIM_SUPPLIER
```

The key columns are as follows.

**FCT_INVENTORY (Inventory Fact)**

| Column Name | Type | Purpose |
|---------|-----|------|
| CLOSING_STOCK | NUMBER | End-of-period inventory quantity |
| REORDER_POINT | NUMBER | Reorder point |
| IS_OUT_OF_STOCK | BOOLEAN | Out-of-stock flag |
| SOLD_QTY | NUMBER | Daily sales quantity |
| DAYS_OF_SUPPLY | NUMBER | Days of supply |

**DIM_CUSTOMER (Customer Master)**

| Column Name | Type | PII |
|---------|-----|-----|
| LAST_NAME / FIRST_NAME | VARCHAR | HIGH |
| BIRTH_DATE | DATE | HIGH |
| GENDER | VARCHAR | MEDIUM |
| AGE_GROUP | VARCHAR | None (recommended alternative column) |

## Metadata Added

### 1. Column Comments (COMMENT ON COLUMN)

Business rules that cannot be determined from column names alone are described as comments.

```sql
-- Explicitly document inventory replenishment logic in comments
COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.FCT_INVENTORY.CLOSING_STOCK
    IS '期末在庫数量。在庫補充判断の基準値。REORDER_POINTと比較して補充要否を判定';

COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.FCT_INVENTORY.REORDER_POINT
    IS '発注点。CLOSING_STOCKがこの値を下回ったら補充発注が必要';

COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.FCT_INVENTORY.IS_OUT_OF_STOCK
    IS '欠品フラグ。TRUE=欠品中（最優先で補充が必要）';

COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.FCT_INVENTORY.SOLD_QTY
    IS '当日の販売数量。売れ筋・人気商品分析に使用する中心的な指標';

-- Clearly state PII classification and usage restrictions for PII columns
COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER.BIRTH_DATE
    IS '生年月日。[PII: HIGH] 個人識別情報。直接使用禁止。AGE_GROUPカラムを代わりに使用すること';

COMMENT ON COLUMN RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER.AGE_GROUP
    IS '年齢層（10代/20代...）。PIIフリーの代替カラム。分析推奨';
```

Comments are stored in `INFORMATION_SCHEMA.COLUMNS`, making them dynamically accessible to AI agents.

```sql
-- Pattern for dynamically retrieving comments when building AI prompts
SELECT
    COLUMN_NAME,
    COMMENT
FROM RETAIL_SAMPLE_DB.INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = 'SALES'
  AND TABLE_NAME   = 'FCT_INVENTORY'
  AND COMMENT IS NOT NULL
ORDER BY ORDINAL_POSITION;
```

Execution result (excerpt):

```
COLUMN_NAME       | COMMENT
------------------+------------------------------------------------------
CLOSING_STOCK     | 期末在庫数量。在庫補充判断の基準値。REORDER_POINTと比較して補充要否を判定
REORDER_POINT     | 発注点。CLOSING_STOCKがこの値を下回ったら補充発注が必要
IS_OUT_OF_STOCK   | 欠品フラグ。TRUE=欠品中（最優先で補充が必要）
SOLD_QTY          | 当日の販売数量。売れ筋・人気商品分析に使用する中心的な指標
```

When the schema changes, you only need to update the comments—no manual modifications to AI prompts are required.

### 2. PII Tags (Object Tags)

Column sensitivity levels are managed as structured data.

```sql
-- Define PII level tags
CREATE TAG IF NOT EXISTS RETAIL_SAMPLE_DB.AI_READY_VERIFY.pii_level
    ALLOWED_VALUES 'HIGH', 'MEDIUM', 'LOW'
    COMMENT = 'PII（個人識別情報）のリスクレベル。HIGH=直接識別子, MEDIUM=準識別子';

-- Apply to DIM_CUSTOMER
ALTER TABLE RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER
    MODIFY COLUMN LAST_NAME  SET TAG RETAIL_SAMPLE_DB.AI_READY_VERIFY.pii_level = 'HIGH';
ALTER TABLE RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER
    MODIFY COLUMN FIRST_NAME SET TAG RETAIL_SAMPLE_DB.AI_READY_VERIFY.pii_level = 'HIGH';
ALTER TABLE RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER
    MODIFY COLUMN BIRTH_DATE SET TAG RETAIL_SAMPLE_DB.AI_READY_VERIFY.pii_level = 'HIGH';
ALTER TABLE RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER
    MODIFY COLUMN GENDER     SET TAG RETAIL_SAMPLE_DB.AI_READY_VERIFY.pii_level = 'MEDIUM';
```

The applied tags can be verified using `TAG_REFERENCES_ALL_COLUMNS`.

```sql
SELECT TAG_NAME, COLUMN_NAME, TAG_VALUE
FROM TABLE(RETAIL_SAMPLE_DB.INFORMATION_SCHEMA.TAG_REFERENCES_ALL_COLUMNS(
    'RETAIL_SAMPLE_DB.SALES.DIM_CUSTOMER', 'table'
))
ORDER BY COLUMN_NAME;
```

```
TAG_NAME  | COLUMN_NAME     | TAG_VALUE
----------+-----------------+----------
PII_LEVEL | BIRTH_DATE      | HIGH
PII_LEVEL | CUSTOMER_ID     | HIGH
PII_LEVEL | EMAIL_OPT_IN    | MEDIUM
PII_LEVEL | FIRST_NAME      | HIGH
PII_LEVEL | GENDER          | MEDIUM
PII_LEVEL | LAST_NAME       | HIGH
PII_LEVEL | PREFECTURE_NAME | MEDIUM
```

### 3. Semantic View (INVENTORY_ANALYSIS)

Table join definitions, synonyms, and SQL generation instructions are centrally managed.

```sql
CREATE OR REPLACE SEMANTIC VIEW RETAIL_SAMPLE_DB.AI_READY_VERIFY.INVENTORY_ANALYSIS

  TABLES (
    inventory AS RETAIL_SAMPLE_DB.SALES.FCT_INVENTORY
      PRIMARY KEY (INVENTORY_KEY)
      WITH SYNONYMS ('在庫データ', '在庫ファクト', '棚卸データ')
      COMMENT = '日次・店舗×商品単位の在庫スナップショット',
    products AS RETAIL_SAMPLE_DB.SALES.DIM_PRODUCT
      PRIMARY KEY (PRODUCT_KEY)
      WITH SYNONYMS ('商品マスタ', '商品', '品目')
      COMMENT = '商品の3階層カテゴリ・原価・定価情報',
    stores AS RETAIL_SAMPLE_DB.SALES.DIM_STORE
      PRIMARY KEY (STORE_KEY)
      WITH SYNONYMS ('店舗マスタ', '店舗', '店')
      COMMENT = '店舗の都道府県・業態情報'
  )

  RELATIONSHIPS (
    inventory_to_products AS inventory (PRODUCT_KEY) REFERENCES products,
    inventory_to_stores   AS inventory (STORE_KEY)   REFERENCES stores
  )

  FACTS (
    inventory.closing_stock AS CLOSING_STOCK
      WITH SYNONYMS ('期末在庫', '手持在庫', '在庫数', '残在庫')
      COMMENT = '期末在庫数量。REORDER_POINTと比較して補充要否を判定',
    inventory.sold_qty AS SOLD_QTY
      WITH SYNONYMS ('販売数量', '売上数量', '売れ数', '販売実績')
      COMMENT = '当日の販売数量。売れ筋分析に使用する中心的な指標',
    inventory.reorder_point AS REORDER_POINT
      WITH SYNONYMS ('発注点', '補充基準')
      COMMENT = '発注点。CLOSING_STOCKがこの値を下回ったら補充発注が必要',
    inventory.is_out_of_stock AS IS_OUT_OF_STOCK
      WITH SYNONYMS ('欠品フラグ', '欠品', '在庫切れ')
      COMMENT = '欠品フラグ。TRUE=欠品中（最優先）'
  )

  DIMENSIONS (
    products.product_name  AS products.PRODUCT_NAME
      WITH SYNONYMS ('商品名', '品名'),
    products.category_l1   AS products.CATEGORY_L1
      WITH SYNONYMS ('大カテゴリ', '商品カテゴリ'),
    products.category_l2   AS products.CATEGORY_L2
      WITH SYNONYMS ('中カテゴリ'),
    stores.store_name      AS stores.STORE_NAME
      WITH SYNONYMS ('店舗名', '店名'),
    stores.prefecture_name AS stores.PREFECTURE_NAME
      WITH SYNONYMS ('都道府県', '地域')
  )

  METRICS (
    inventory.total_sold AS SUM(inventory.sold_qty)
      WITH SYNONYMS ('総販売数量', '合計売上数', '売れ筋指標')
      COMMENT = '合計販売数量。売れ筋商品ランキングに使用',
    inventory.needs_reorder AS SUM(
        CASE WHEN inventory.closing_stock < inventory.reorder_point THEN 1 ELSE 0 END
    )
      WITH SYNONYMS ('補充必要件数', '発注必要数')
      COMMENT = '在庫補充が必要な商品×店舗の件数'
  )

  COMMENT = '在庫分析用セマンティックビュー'
  AI_SQL_GENERATION '在庫補充の判断は closing_stock < reorder_point を使用する。
売れ筋分析には sold_qty の合計（total_sold）を使用する。
IS_OUT_OF_STOCKがTRUEは欠品中で最優先補充対象。
クエリには必ず店舗名と商品名を含め、日本語のカラム別名を付けること。
IS_ACTIVEがTRUEの営業中店舗のみを対象にすること。';
```

## Test 1: Comparing Inventory Replenishment SQL Generation Quality

**Question**: "Write a SQL query to identify products that need inventory replenishment. Include store name and product name."

### Without Metadata (Before)

Only column names were provided to the AI.

```
FCT_INVENTORY(INVENTORY_KEY, DATE_KEY, STORE_KEY, PRODUCT_KEY,
              OPENING_STOCK, RECEIVED_QTY, SOLD_QTY, CLOSING_STOCK,
              IS_OUT_OF_STOCK, REORDER_POINT, ORDER_QUANTITY, ...)
DIM_PRODUCT(PRODUCT_KEY, PRODUCT_NAME, CATEGORY_L1, ...)
DIM_STORE(STORE_KEY, STORE_NAME, PREFECTURE_NAME, ...)
```

Generated SQL:

```sql
SELECT DISTINCT
    DIM_STORE.STORE_NAME,
    DIM_PRODUCT.PRODUCT_NAME
FROM FCT_INVENTORY
JOIN DIM_PRODUCT ON FCT_INVENTORY.PRODUCT_KEY = DIM_PRODUCT.PRODUCT_KEY
JOIN DIM_STORE   ON FCT_INVENTORY.STORE_KEY   = DIM_STORE.STORE_KEY
WHERE FCT_INVENTORY.CLOSING_STOCK < FCT_INVENTORY.REORDER_POINT
  AND FCT_INVENTORY.IS_OUT_OF_STOCK = 0;
```

### With Metadata (After)

Column comments and the Semantic View's `AI_SQL_GENERATION` instructions were included in the prompt.

Generated SQL:

```sql
SELECT
    DIM_STORE.STORE_NAME          AS "店舗名",
    DIM_PRODUCT.PRODUCT_NAME      AS "商品名",
    FCT_INVENTORY.CLOSING_STOCK   AS "期末在庫",
    FCT_INVENTORY.REORDER_POINT   AS "発注点",
    FCT_INVENTORY.IS_OUT_OF_STOCK AS "欠品フラグ",
    FCT_INVENTORY.DAYS_OF_SUPPLY  AS "在庫日数"
FROM FCT_INVENTORY
JOIN DIM_STORE   ON FCT_INVENTORY.STORE_KEY   = DIM_STORE.STORE_KEY
JOIN DIM_PRODUCT ON FCT_INVENTORY.PRODUCT_KEY = DIM_PRODUCT.PRODUCT_KEY
WHERE DIM_STORE.IS_ACTIVE = TRUE
  AND FCT_INVENTORY.CLOSING_STOCK < FCT_INVENTORY.REORDER_POINT
ORDER BY FCT_INVENTORY.IS_OUT_OF_STOCK DESC,
         FCT_INVENTORY.DAYS_OF_SUPPLY  ASC
LIMIT 20;
```

I confirmed that the query executed successfully and returned 10 rows of data.

### Difference Summary

| Aspect | Before | After |
|------|--------|-------|
| Excluding closed stores | None (all stores included) | Excluded with `IS_ACTIVE = TRUE` |
| Sort order | None | Out-of-stock flag descending → Days of supply ascending (by urgency) |
| Row limit | None (all rows) | `LIMIT 20` |
| Column aliases | English only | Japanese aliases ("店舗名", "商品名", etc.) |
| IS_OUT_OF_STOCK interpretation | `= 0` (unclear condition) | Correctly used as `DESC` sort key |

## Test 2: Comparing Responses to the Ambiguous Query "Best-Selling"

**Question**: "Show me a category-wise ranking of best-selling products."

### Without Metadata (Before)

```sql
SELECT
    DIM_PRODUCT.CATEGORY_L1,
    DIM_PRODUCT.CATEGORY_L2,
    SUM(FCT_INVENTORY.SOLD_QTY) AS TOTAL_SOLD_QTY
FROM DIM_PRODUCT
JOIN FCT_INVENTORY ON DIM_PRODUCT.PRODUCT_KEY = FCT_INVENTORY.PRODUCT_KEY
GROUP BY DIM_PRODUCT.CATEGORY_L1, DIM_PRODUCT.CATEGORY_L2
ORDER BY TOTAL_SOLD_QTY DESC;
```

### With Metadata (After)

The Semantic View's synonyms (`売れ筋指標 = total_sold`) and `AI_SQL_GENERATION` instructions guided the AI in the right direction.

```sql
SELECT
    p.CATEGORY_L1    AS "大カテゴリ",
    p.PRODUCT_NAME   AS "商品名",
    SUM(i.SOLD_QTY)  AS "総販売数量"
FROM FCT_INVENTORY i
JOIN DIM_PRODUCT p ON i.PRODUCT_KEY = p.PRODUCT_KEY
GROUP BY 1, 2
QUALIFY RANK() OVER (PARTITION BY p.CATEGORY_L1 ORDER BY SUM(i.SOLD_QTY) DESC) <= 3
ORDER BY "大カテゴリ", "総販売数量" DESC;
```

### Difference Summary

| Aspect | Before | After |
|------|--------|-------|
| Interpretation of "best-selling" | Guessed `SOLD_QTY` | Explicitly resolved via synonyms: `SOLD_QTY = 販売実績 = 売れ筋指標` |
| Ranking method | Simple `ORDER BY DESC` across all data | `QUALIFY RANK() OVER (PARTITION BY CATEGORY_L1)` for within-category ranking |
| Category axis | Grouped by both L1 and L2 | L1 only (appropriately scoped based on `AI_SQL_GENERATION` instructions) |
| Column aliases | English | Japanese |

The Before SQL can only rank by overall totals and fails to meet the "by category" requirement. The After SQL extracts the top 3 within each major category.

## Test 3: Comparing Safety with PII-Related Queries

**Question**: "Write a SQL query to analyze the list of personal information and point balances by membership rank."

### Without Metadata (Before)

```sql
SELECT
    MEMBER_RANK,
    COUNT(CUSTOMER_KEY) AS CUSTOMER_COUNT,
    AVG(AGE_GROUP)      AS AVERAGE_AGE,
    AVG(POINT_BALANCE)  AS AVERAGE_POINT_BALANCE,
    SUM(POINT_BALANCE)  AS TOTAL_POINT_BALANCE
FROM DIM_CUSTOMER
GROUP BY MEMBER_RANK
ORDER BY MEMBER_RANK;
```

The result happened to be an aggregate query that avoided direct exposure of personal information. However, this was a spontaneous decision by the AI without explicit justification. Depending on how the prompt is worded or how specific the question is, there is a risk that `LAST_NAME` or `BIRTH_DATE` could be directly included in the SELECT clause.

### With Metadata (After)

PII tags and column comments were included in the prompt.

```sql
SELECT
    MEMBER_RANK,
    COUNT(CUSTOMER_KEY)    AS CUSTOMER_COUNT,
    SUM(POINT_BALANCE)     AS TOTAL_POINT_BALANCE,
    AVG(POINT_BALANCE)     AS AVERAGE_POINT_BALANCE,
    MIN(POINT_BALANCE)     AS MIN_POINT_BALANCE,
    MAX(POINT_BALANCE)     AS MAX_POINT_BALANCE
FROM DIM_CUSTOMER
WHERE MEMBER_RANK IN ('レギュラー', 'シルバー', 'ゴールド', 'プラチナ')
GROUP BY MEMBER_RANK;
```

### Difference Summary

| Aspect | Before | After |
|------|--------|-------|
| Handling of HIGH PII columns | Implicitly avoided | Explicitly excluded based on tag evidence |
| `BIRTH_DATE` alternative | Implicitly unused | Unused per comment instruction "Use `AGE_GROUP` instead" |
| Basis for safety controls | AI's probabilistic judgment | Structured data via PII tags (HIGH/MEDIUM) |
| Reproducibility | Low (changes with prompt variations) | High (based on metadata persisted in the database) |
| MEMBER_RANK values | Unknown (generic query) | Values retrieved from comments (Regular/Silver/Gold/Platinum) and reflected in `IN` clause |

The Before result appearing safe is merely coincidental and cannot be considered stable control.

## Dynamic Context Building: Why Store Metadata in the Database

Metadata is persisted in the database and can be dynamically retrieved by AI agents at query time.

```sql
-- Dynamically retrieve comments to build the context portion of AI prompts
SELECT
    'COLUMN ' || COLUMN_NAME || ': ' || COMMENT AS column_context
FROM RETAIL_SAMPLE_DB.INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = 'SALES'
  AND TABLE_NAME   = 'FCT_INVENTORY'
  AND COMMENT IS NOT NULL
ORDER BY ORDINAL_POSITION;
```

```
column_context
---------------------------------------------------------------------
COLUMN OPENING_STOCK: 期首在庫数量（その日の開始時点の在庫）
COLUMN SOLD_QTY: 当日の販売数量。売れ筋・人気商品分析に使用する中心的な指標
COLUMN CLOSING_STOCK: 期末在庫数量。在庫補充判断の基準値。REORDER_POINTと比較して補充要否を判定
COLUMN IS_OUT_OF_STOCK: 欠品フラグ。TRUE=欠品中（最優先で補充が必要）
COLUMN REORDER_POINT: 発注点。CLOSING_STOCKがこの値を下回ったら補充発注が必要
```

By passing this output to the `SNOWFLAKE.CORTEX.COMPLETE` prompt, the latest information automatically reaches the AI even when table definitions change. There is no need to manually rewrite prompts.

```
[Schema change] → [Update comments] → [Automatically reflected in next AI call]
                                        ↑ No prompt rewriting needed here
```

## Conclusion

Here is a summary of what was confirmed through the three tests.

| Aspect | Without Metadata | With Metadata |
|------|--------------|--------------|
| Reflecting business rules | Relies on guessing from column names | Accurately retrieved from comments |
| Handling ambiguous queries | Limited to simple aggregation | Intent resolved via synonyms and AI_SQL_GENERATION instructions |
| PII control stability | Probabilistic and prompt-dependent | Stable control based on structured tag data |
| Adapting to schema changes | Requires manual prompt modifications | Dynamically retrieved from INFORMATION_SCHEMA |

In terms of implementation priority, the following order felt most practical.

1. **Column comments** — No impact on existing tables, lowest cost with the most immediate effect
2. **Object Tags (PII levels)** — Prioritize when security or compliance requirements exist
3. **Semantic Views** — Create when you're ready to build a full-fledged AI interface

Maintaining metadata is similar to "writing documentation," but the fact that it directly influences AI agent behavior means a higher ROI can be expected compared to traditional documentation efforts.
