## はじめに

本記事では、Claude Codeの[スキル機能](https://code.claude.com/docs/en/skills)を活用し、Hugoブログ記事の執筆ワークフロー全体を自動化した構成についてまとめる。「記事を書いて」の一言で、構成プランニングから本文執筆、4段の品質チェック、デプロイまでを9つのスキルが連携して処理する仕組みを構築した。

## Claude Codeのスキル機能の概要

Claude Codeのスキルは、`.claude/skills/<skill-name>/SKILL.md` を配置することでClaudeの機能を拡張する仕組みである。YAMLフロントマターとMarkdown形式の指示文を記述するだけで、以下のような動作が可能になる。

| 特徴 | 説明 |
|------|------|
| 自動トリガー | `description` に記載したキーワードに合致する会話でClaudeが自動的にスキルを読み込む |
| 手動実行 | `/skill-name` のスラッシュコマンドで明示的に起動できる |
| サブエージェント連携 | `context: fork` の指定でサブエージェント内で隔離実行できる |
| ツール制限 | `allowed-tools` で使用可能なツールを限定できる |
| サポートファイル | SKILL.md以外のファイル（テンプレート、リファレンス、スクリプト等）を同一ディレクトリに格納し参照できる |

スキル間の連携も可能で、あるスキルの指示文からTask toolを使って別のスキルを起動するパターンで、複雑なワークフローを構築できる。

## ワークフロー全体像

構築したワークフローは、9つのスキルが以下の順序で連携動作する。

```
┌─────────────────────────────────────────────────────────────┐
│  ユーザー: 「〇〇の記事を書いて」                              │
└─────────────┬───────────────────────────────────────────────┘
              ▼
┌─────────────────────────┐
│ 1. article-planner      │ ← 構成プランニング（参考情報収集・構成案生成）
│    [ユーザー承認ゲート]   │
└─────────────┬───────────┘
              ▼
┌─────────────────────────┐
│ 2. hugo-blog-post       │ ← 記事本文の執筆・ファイル出力
│    + chart-visualizer   │   （図表が必要な場合のみ）
└─────────────┬───────────┘
              ▼
┌─────────────────────────┐
│ 3. tag-icon-generator   │ ← 未登録タグのSVGアイコン生成
└─────────────┬───────────┘
              ▼
┌─────────────────────────┐
│ 4. 品質チェック（順次）   │
│  4-1. ai-writing-checker │ ← AIっぽさ検出・自動修正
│  4-2. blog-fact-checker  │ ← ファクトチェック・自動修正
│  4-3. seo-checker        │ ← SEO検証・summary自動生成
│  4-4. internal-link-     │ ← 内部リンク提案
│       checker            │
└─────────────┬───────────┘
              ▼
┌─────────────────────────┐
│ 5. hugo-deploy          │ ← ビルド＋GitHub Pagesデプロイ
│    （ユーザー指示時のみ） │
└─────────────────────────┘
```

各スキルの役割を整理する。

| # | スキル名 | 役割 | 実行タイミング |
|---|----------|------|--------------|
| 1 | `hugo-blog-post` | メインオーケストレーター。全体の制御フロー定義 | ユーザー指示で起動 |
| 2 | `article-planner` | トピック分析、参考情報収集、構成案生成 | Step 2で自動起動 |
| 3 | `chart-visualizer` | Plotlyチャート画像の生成 | 構成案に図の指示がある場合 |
| 4 | `tag-icon-generator` | タグ用SVGアイコンの取得/生成 | 記事出力後に自動起動 |
| 5 | `ai-writing-checker` | AIっぽさの検出と自動修正 | 品質チェック第1段 |
| 6 | `blog-fact-checker` | 技術的正確性の検証と修正 | 品質チェック第2段 |
| 7 | `seo-checker` | SEOメタ情報の検証とsummary生成 | 品質チェック第3段 |
| 8 | `internal-link-checker` | 既存記事との内部リンク提案 | 品質チェック第4段 |
| 9 | `hugo-deploy` | Hugoビルドとデプロイ | ユーザー指示時のみ |

## 各スキルの設計と実装

### hugo-blog-post（オーケストレーター）

全体のワークフローを制御するメインスキルである。SKILL.mdに以下の情報を定義している。

- **プロジェクト情報**: 記事格納先のパス、フォルダ命名規則
- **フロントマターテンプレート**: Hugo記事のYAMLヘッダー
- **文体ルール**: 常体（だ・である調）、絵文字禁止、簡潔記述等の制約
- **ショートコード**: リンクカード、折りたたみ、強調ボックスの書式
- **実行手順**: Step 1〜9の詳細な手順と各スキルの呼び出し方法

他のスキルはすべてこのオーケストレーターの指示文内でTask tool（サブエージェント）として起動される。メインのコンテキストウィンドウを消費せずに各チェック処理を実行できる点が利点である。

### article-planner（構成プランニング）

記事執筆前の構成プランニングを担当する。トピックを受け取ると以下を実行する。

1. **トピック分析**: スコープの明確化と想定読者レベルの推定
2. **参考情報収集**: Snowflakeトピックの場合はSnowflake Docs MCPサーバーで検索、その他はWebSearchで公式ドキュメントを収集
3. **構成案生成**: メタ情報テーブル（タイトル候補、URLスラッグ、推奨タグ）とセクション構成を出力
4. **ユーザー承認**: 構成案を提示し、承認を得てから次へ進む

ここで生成された「参考ドキュメントURL一覧」と「検証すべき技術的主張リスト」は、後続のファクトチェックに引き渡される。同じ情報を二重に検索することを防ぐ設計である。

### 品質チェックパイプライン

記事出力後、4段のチェックを**順次実行**する。並列ではなく順次にしているのは、各チェッカーが同一ファイルを読み書きするため、ファイル競合を防ぐ必要があるためである。

**ai-writing-checker（AIっぽさチェック）**

正規表現パターンと構造分析の2段構成で、AI生成文章の特徴を検出する。

| チェック種別 | 内容 |
|------------|------|
| 正規表現パターン | です/ます体の混入、「ぜひ」「いかがでしたでしょうか」等のAI定番表現を検出 |
| 構造分析 | 文末パターンの多様性、はじめに/まとめの構造、見出し数、論理展開を評価 |
| バースティネス | 文長の標準偏差を算出し、均一すぎる場合に警告 |

検出結果を0-100のAIっぽさスコアで定量化し、CRITICAL/WARNING/INFOの3段階で報告する。CRITICALとWARNING（機械的修正可能なもの）はEdit toolで自動修正する。

**blog-fact-checker（ファクトチェック）**

記事内の技術的主張を公式ドキュメントと突き合わせて検証する。

| 検証項目 | 方法 |
|---------|------|
| 技術的正確性 | Snowflake Docs MCPで公式ドキュメントを検索・照合 |
| SQL構文 | 構文リファレンスとの突き合わせ、括弧/引用符の対応確認 |
| URL有効性 | WebFetchによるアクセス確認（404検出、リダイレクト検出） |
| バージョン・日付 | WebSearchで最新バージョンとの整合性確認 |

article-plannerから引き渡された参考URL一覧を活用し、検索の重複を回避する。

**seo-checker（SEOチェック）**

SEOメタ情報の検証と、`summary`フィールドの自動生成を行う。

| チェック項目 | 基準 |
|------------|------|
| summary | 空の場合、記事本文から100-160文字の概要を自動生成 |
| タイトル長 | 41-59文字が推奨。範囲外は警告 |
| URLスラッグ | 英語・ハイフン区切りであること、30文字以内 |
| og:image | featured.jpg/pngの存在確認 |

**internal-link-checker（内部リンクチェック）**

新規記事のタグ・キーワードと既存記事を照合し、リンク追加の提案を行う。最新100記事のフロントマターを走査し、タグが一致する記事をマッチングする。自動修正は行わず提案のみ出力する設計としている。

### 補助スキル

**tag-icon-generator**は、記事のフロントマターに含まれるタグにアイコンが未登録の場合、Simple Icons CDNからSVGを取得する。CDNに存在しないタグにはテキストベースのフォールバックSVGを生成し、Hugoテンプレート（`tag-icon.html` / `tag-badge-icon.html`）のマッピングも自動更新する。

**hugo-deploy**は、`hugo --config hugo.toml -d docs` でビルドし、git commit + pushでGitHub Pagesにデプロイする。ユーザーが明示的に指示した場合のみ実行する。

## スキル間連携の設計パターン

このワークフローで採用している設計パターンを整理する。

**データの引き渡しによる重複排除**

article-plannerが収集した「参考ドキュメントURL一覧」と「検証すべき技術的主張リスト」をblog-fact-checkerに引き渡す。fact-checkerは引き渡された情報をまず確認し、不足分のみ新規検索する。同じドキュメントを2回取得する無駄を省いている。

**順次実行による競合防止**

品質チェック4段は順次実行とした。各チェッカーがEdit toolで同一の `index.md` を修正するため、並列実行するとファイルの読み書きが衝突する。Step 9-1（AIっぽさチェック）の完了を待ってからStep 9-2（ファクトチェック）を起動し、以降も同様に順次実行する。

**サブエージェントへの委譲**

品質チェックの各ステップはTask tool（`subagent_type="general-purpose"`）で起動する。メインのコンテキストウィンドウにチェック結果の詳細が蓄積されることを防ぎ、各チェッカーの処理が完了した時点で結果のサマリーだけが返る。

**ユーザー承認ゲート**

article-plannerの構成案提示後にユーザー承認を挟む。タイトル、URLスラッグ、タグ、セクション構成を確認してから本文生成に進むことで、方向性のずれを早期に修正できる。

## まとめ

Claude Codeのスキル機能を活用して、ブログ記事の執筆ワークフロー全体を9つのスキルの連携で自動化した。構成プランニング、記事執筆、タグアイコン生成、4段の品質チェック（AIっぽさ、ファクト、SEO、内部リンク）、デプロイまでを「記事を書いて」の一言でトリガーできる。スキル間のデータ引き渡しによる重複排除、順次実行による競合防止、サブエージェントへの委譲によるコンテキスト保全が設計上のポイントである。
